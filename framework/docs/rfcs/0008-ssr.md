# RFC 0008 ‚Äî Server-side rendering (`@zoijs/ssr`)

- Status: **Accepted** ‚Äî implemented as `@zoijs/ssr` 0.1.0 + a DOM-free core compiler
  in `@zoijs/core` 1.5.0 (2026-06-26)
- Target: a new **optional package** (`@zoijs/ssr`, peer on `@zoijs/core`, zero deps)
  plus an additive, DOM-free refactor of the core template compiler and a new
  `@zoijs/core/server` subpath.
- Affects: **no change to the client runtime's behavior** (byte-identical output); the
  learnable nine-function main surface is unchanged. Phase 8 ‚Äî the largest *optional*
  milestone.

## 1. Problem statement

Zoijs renders in the browser: `html\`‚Ä¶\`` builds a `<template>` element, the renderer
clones it and wires fine-grained bindings. That's great for a SPA, but it leaves two
real needs unmet:

1. **SEO / first paint** ‚Äî a crawler or a slow device gets a blank `<div id="app">`
   until JS runs.
2. **Static sites** ‚Äî the docs site already prerenders routes to flat HTML, but it has
   to spin up **jsdom** to do it, because there's no way to render a component to a
   string without a DOM.

The blocker is structural: `html()` calls `createTemplate()` (a `document.createElement`)
**eagerly**, so merely *calling a component* on a server throws. There is no DOM-free
path to a component's markup.

## 2. Rule of Three

1. **Explainable in under two minutes?** **Yes.** "Render a component to an HTML string
   on the server; the same component `mount`s on the client." `renderToString(App)` ‚Üí
   markup. No new concepts in the view.
2. **Implementable without core changes?** **Almost.** The string renderer itself is a
   zero-dep package. But it needs two things from the core: a component must be
   *evaluable without a DOM*, and the renderer must reuse the *exact* security
   predicates (escaping, URL schemes) so server and client never diverge. Both are
   **additive** core changes (a DOM-free compiler + a `@zoijs/core/server` subpath) that
   don't alter client behavior ‚Äî the standard RFC-gated bar.
3. **Would 80‚Äì90% of apps benefit?** **The ones that need SEO or static output, yes** ‚Äî
   a large and important slice (marketing sites, docs, content). It's explicitly an
   **opt-in** package; a pure SPA ignores it and pays nothing. It clears the bar as the
   ROADMAP's big optional milestone.

**Verdict: ship** ‚Äî as an additive core refactor + an optional package.

## 3. Design

### 3.1 Core: a DOM-free compiler (additive, 1.5.0)

`compile(strings)` already builds the static HTML string + part descriptors with **no
DOM**; the only DOM touch was the final `createTemplate(out)`. The change:

- `compile()` returns the static HTML string + parts (no DOM).
- `html()` returns a result whose `.template` (the `<template>` element) is created
  **lazily on first client render**, and which carries the static HTML (`__staticHTML`)
  and a `__zoijsTemplate` brand. So `html()` touches no DOM; a server can evaluate a
  component to a result and read its skeleton.
- A new subpath **`@zoijs/core/server`** exposes the building blocks a string renderer
  needs: `isTemplateResult` / `isEachMarker` / `templateHTML`, and the **same** security
  predicates the client uses ‚Äî `escapeText`, `escapeAttr`, `isSafeUrl`,
  `isSafeAttributeName`, `URL_ATTRS` (the last two/three moved into `security.js` so
  there is a single source of truth shared by both renderers).

Client rendering is unchanged (the lazy `<template>` is created exactly when it was
before ‚Äî at render time ‚Äî and cached). Verified: 150 unit + 73 browser tests still pass.

### 3.2 `@zoijs/ssr`: the string renderer

`renderToString(component)` walks the static HTML skeleton and injects values at its two
markers (`<!--zoijs-->` for a child slot, ` data-zoijs-bind` on a dynamic element),
consuming the document-ordered `parts` in lockstep. For each slot it reads the value
**once** and serializes it:

- **Text** ‚Üí `escapeText`. **Attributes** ‚Üí `escapeAttr`, with the client renderer's
  exact branching (`value`/`checked` ‚Üí markup form, `false`/`null` ‚Üí omit, `true` ‚Üí
  bare, URL attrs ‚Üí `isSafeUrl` or drop, unsafe names ‚Üí drop). **Events and `ref`s** ‚Üí
  dropped (wired on the client). **Nested templates / `each` lists / arrays / falsy** ‚Üí
  recurse / iterate / render nothing ‚Äî the same value semantics as a text binding.

Zero dependencies, no DOM. The output is the component's markup; the user places it in
their HTML shell.

## 4. Hydration ‚Äî what ships now, and what's deferred

**Ships now:** server render for first paint + SEO/SSG, and the client `mount`s over it
(builds its own DOM and takes over). This satisfies the milestone's "Done when": the
same component code runs on server and client, and **removing `@zoijs/ssr` leaves a
working client app** (the client path is pure `@zoijs/core`).

**Deferred (recorded):** *seamless* hydration ‚Äî the client **adopting** the server's DOM
nodes in place instead of re-creating them. This is genuinely a core change, not a
package: the renderer's text and list bindings *create* nodes (a text binding inserts a
fresh `Text` node; `each` builds items), so adopting existing server nodes requires
**hydration-aware bindings** throughout `renderer.js`. That's a substantial,
behavior-sensitive core effort ‚Äî a future RFC against the core, deliberately not rushed
into this package. The string renderer and the `@zoijs/core/server` seam it's built on
are exactly the foundation that future work will reuse.

## 5. Security

Server output goes through the **same** predicates as the client (shared from
`security.js`), so there is no second escaping implementation to drift. The Phase 2 XSS
vectors are re-tested against the string output: `javascript:`/control-char URLs dropped,
`data:text/html` dropped (raster `data:image` kept), `<script>` in text escaped,
attribute breakout escaped, `on*`/`srcdoc` refused, handlers/`ref`s never serialized.
Re-runs the supply-chain gate: `@zoijs/ssr` has zero deps and a single `@zoijs/core`
peer.

## 6. Out of scope (guardrails)

- **No client build step, ever.** SSR is server-only and opt-in; the client path is
  untouched and still runs from `<script type="module">`.
- **No streaming / islands / partial hydration / ISR** ‚Äî none has earned its place; each
  would be its own future RFC if it ever does.
- **No SSR-only component API.** The same `html`/`each`/components render in both places;
  there is no server-only variant to learn.
- **No raw-DOM serialization** ‚Äî a component returning a real `Node` throws on the server
  (there is no DOM); return `html\`‚Ä¶\`` instead.

## 7. Alternatives considered

- **jsdom/linkedom on the server** (run the real client renderer against a fake DOM).
  Behavior-identical for free, and what the docs site does today ‚Äî but it adds a runtime
  dependency to a package, breaking the zero-dep star topology, for output a ~120-line
  string renderer produces directly. Rejected in favor of the DOM-free path (which also
  lets the docs site drop jsdom).
- **A separate server template syntax / compiler.** Duplicates the parser and invites
  drift in escaping ‚Äî exactly the security risk we avoid by sharing `security.js`.
- **Shipping seamless hydration now.** The valuable, hard part; but it needs
  hydration-aware core bindings (¬ß4). Bundling a half-working version would be worse than
  a clear, correct "mount over server HTML" with the seam in place for later.

## 8. Decision

**Accepted.** Land the DOM-free compiler + `@zoijs/core/server` subpath in `@zoijs/core`
1.5.0 (additive; client byte-identical), and ship `@zoijs/ssr` 0.1.0 with
`renderToString`. Add tests (DOM-free server render, security parity, and
server‚Üîclient parity), types, a README, a docs page, and the ecosystem listings; wire
into the supply-chain, doc-coverage, and root test gates. Seamless DOM-adopting
hydration is a deferred future core RFC.
