# RFC 0008 — Server-side rendering (`@zoijs/ssr`)

- Status: **Accepted** — `renderToString` + DOM-free compiler in `@zoijs/core` 1.5.0
  / `@zoijs/ssr` 0.1.0 (2026-06-26); **in-place hydration** in `@zoijs/core` 1.6.0 /
  `@zoijs/ssr` 0.2.0 (2026-06-27, see §4)
- Target: a new **optional package** (`@zoijs/ssr`, peer on `@zoijs/core`, zero deps)
  plus an additive, DOM-free refactor of the core template compiler and a new
  `@zoijs/core/server` subpath.
- Affects: **no change to the client runtime's behavior** (byte-identical output); the
  learnable nine-function main surface is unchanged. Phase 8 — the largest *optional*
  milestone.

## 1. Problem statement

Zoijs renders in the browser: `html\`…\`` builds a `<template>` element, the renderer
clones it and wires fine-grained bindings. That's great for a SPA, but it leaves two
real needs unmet:

1. **SEO / first paint** — a crawler or a slow device gets a blank `<div id="app">`
   until JS runs.
2. **Static sites** — the docs site already prerenders routes to flat HTML, but it has
   to spin up **jsdom** to do it, because there's no way to render a component to a
   string without a DOM.

The blocker is structural: `html()` calls `createTemplate()` (a `document.createElement`)
**eagerly**, so merely *calling a component* on a server throws. There is no DOM-free
path to a component's markup.

## 2. Rule of Three

1. **Explainable in under two minutes?** **Yes.** "Render a component to an HTML string
   on the server; the same component `mount`s on the client." `renderToString(App)` →
   markup. No new concepts in the view.
2. **Implementable without core changes?** **Almost.** The string renderer itself is a
   zero-dep package. But it needs two things from the core: a component must be
   *evaluable without a DOM*, and the renderer must reuse the *exact* security
   predicates (escaping, URL schemes) so server and client never diverge. Both are
   **additive** core changes (a DOM-free compiler + a `@zoijs/core/server` subpath) that
   don't alter client behavior — the standard RFC-gated bar.
3. **Would 80–90% of apps benefit?** **The ones that need SEO or static output, yes** —
   a large and important slice (marketing sites, docs, content). It's explicitly an
   **opt-in** package; a pure SPA ignores it and pays nothing. It clears the bar as the
   ROADMAP's big optional milestone.

**Verdict: ship** — as an additive core refactor + an optional package.

## 3. Design

### 3.1 Core: a DOM-free compiler (additive, 1.5.0)

`compile(strings)` already builds the static HTML string + part descriptors with **no
DOM**; the only DOM touch was the final `createTemplate(out)`. The change:

- `compile()` returns the static HTML string + parts (no DOM).
- `html()` returns a result whose `.template` (the `<template>` element) is created
  **lazily on first client render**, and which carries the static HTML (`__staticHTML`)
  and a `__zoijsTemplate` brand. So `html()` touches no DOM; a server can evaluate a
  component to a result and read its skeleton.
- A new subpath **`@zoijs/core/server`** exposes the building blocks a string renderer
  needs: `isTemplateResult` / `isEachMarker` / `templateHTML`, and the **same** security
  predicates the client uses — `escapeText`, `escapeAttr`, `isSafeUrl`,
  `isSafeAttributeName`, `URL_ATTRS` (the last two/three moved into `security.js` so
  there is a single source of truth shared by both renderers).

Client rendering is unchanged (the lazy `<template>` is created exactly when it was
before — at render time — and cached). Verified: 150 unit + 73 browser tests still pass.

### 3.2 `@zoijs/ssr`: the string renderer

`renderToString(component)` walks the static HTML skeleton and injects values at its two
markers (`<!--zoijs-->` for a child slot, ` data-zoijs-bind` on a dynamic element),
consuming the document-ordered `parts` in lockstep. For each slot it reads the value
**once** and serializes it:

- **Text** → `escapeText`. **Attributes** → `escapeAttr`, with the client renderer's
  exact branching (`value`/`checked` → markup form, `false`/`null` → omit, `true` →
  bare, URL attrs → `isSafeUrl` or drop, unsafe names → drop). **Events and `ref`s** →
  dropped (wired on the client). **Nested templates / `each` lists / arrays / falsy** →
  recurse / iterate / render nothing — the same value semantics as a text binding.

Zero dependencies, no DOM. The output is the component's markup; the user places it in
their HTML shell.

## 4. Hydration — in-place adoption (shipped, core 1.6.0 / ssr 0.2.0)

The client **adopts** the server DOM in place instead of re-creating it.

- **API.** Additive: `mount(component, target, { hydrate: true })` (no new export — the
  nine-function surface is unchanged), re-exported as `@zoijs/ssr`'s `hydrate()`. The
  server marks its output with `renderToString(component, { hydratable: true })`.
- **What's adopted exactly.** Element structure, attributes, and events. `render`, given
  a hydrate root, walks the *existing* server subtree (no clone); `applyAttribute` is
  idempotent so reactive attributes just re-apply the server value, and events simply
  `addEventListener` to the live element. The expensive, layout-bearing part of the page
  is reused byte-for-byte — same node identity.
- **What re-renders (invisibly).** Each dynamic *child slot's* content. `renderToString`
  (hydratable) brackets each top-level slot with a `<!--zoijs:[-->` start marker + the
  `<!--zoijs-->` anchor; on hydrate, `render` clears exactly that slot (back to its start
  marker, preserving static text) and the normal binding renders fresh content there.
  Because the values match the server, there is no visible change and no flash.
- **Why this shape.** HTML serialize→parse merges adjacent text, so the server can't
  reproduce the client's exact per-slot text-node boundaries. Re-rendering the dynamic
  *leaves* into the adopted structure sidesteps that entirely and keeps the model uniform
  across text, conditionals, nested templates, and `each` lists — at the cost of
  re-creating only the dynamic content (cheap, invisible), never the page shell. Markers
  are emitted for the **outermost** template only; nested content is rendered clean (it's
  cleared and re-rendered anyway). On a structural mismatch the binding simply doesn't
  attach for that region — graceful degradation, never corruption.

**Still deferred:** *exact* node-by-node adoption (zero re-render of dynamic leaves /
keyed-list items), which would need per-slot and per-item boundary markers and keyed-list
adoption. The in-place approach delivers the user-visible SSR value (no full re-create,
no flash, SEO, interactive) at far lower risk; exact adoption can build on this seam.

### 4.1 Follow-ons (shipped)

- **SSR-safe optional packages** (`@zoijs/head` 0.1.1, `@zoijs/router` 0.2.1). Both
  guarded their `window`/`document` access so a routed page that sets `<title>` can be
  passed to `renderToString` without throwing: `head`'s setters are no-ops server-side
  (put the initial tags in the HTML shell), and `router`'s `view()` renders the matched
  route's template directly (the `"/"` route, since no request URL is available) while
  navigation listeners are skipped. No API change.
- **Server→client data hand-off** (`@zoijs/resource` 0.2.0, `@zoijs/ssr` 0.3.0).
  `renderToString` is synchronous, so a resource renders its loading state on the server.
  `resource(fetcher, { initial })` starts already-settled with a value and skips the
  auto-load; `@zoijs/ssr`'s `serialize(value)` emits JSON safe to embed in a `<script>`
  (escapes `<`, `>`, `&`, U+2028/U+2029). The server renders with the data and embeds
  `serialize(data)`; the client seeds the resource with it — no refetch, no flash. This
  is the *primitive*; per-request routed SSR and data loaders remain deferred future work.

## 5. Security

Server output goes through the **same** predicates as the client (shared from
`security.js`), so there is no second escaping implementation to drift. The Phase 2 XSS
vectors are re-tested against the string output: `javascript:`/control-char URLs dropped,
`data:text/html` dropped (raster `data:image` kept), `<script>` in text escaped,
attribute breakout escaped, `on*`/`srcdoc` refused, handlers/`ref`s never serialized.
Re-runs the supply-chain gate: `@zoijs/ssr` has zero deps and a single `@zoijs/core`
peer.

## 6. Out of scope (guardrails)

- **No client build step, ever.** SSR is server-only and opt-in; the client path is
  untouched and still runs from `<script type="module">`.
- **No streaming / islands / partial hydration / ISR** — none has earned its place; each
  would be its own future RFC if it ever does.
- **No SSR-only component API.** The same `html`/`each`/components render in both places;
  there is no server-only variant to learn.
- **No raw-DOM serialization** — a component returning a real `Node` throws on the server
  (there is no DOM); return `html\`…\`` instead.

## 7. Alternatives considered

- **jsdom/linkedom on the server** (run the real client renderer against a fake DOM).
  Behavior-identical for free, and what the docs site does today — but it adds a runtime
  dependency to a package, breaking the zero-dep star topology, for output a ~120-line
  string renderer produces directly. Rejected in favor of the DOM-free path (which also
  lets the docs site drop jsdom).
- **A separate server template syntax / compiler.** Duplicates the parser and invites
  drift in escaping — exactly the security risk we avoid by sharing `security.js`.
- **Shipping seamless hydration now.** The valuable, hard part; but it needs
  hydration-aware core bindings (§4). Bundling a half-working version would be worse than
  a clear, correct "mount over server HTML" with the seam in place for later.

## 8. Decision

**Accepted.** Land the DOM-free compiler + `@zoijs/core/server` subpath in `@zoijs/core`
1.5.0 (additive; client byte-identical), and ship `@zoijs/ssr` 0.1.0 with
`renderToString`. Add tests (DOM-free server render, security parity, and
server↔client parity), types, a README, a docs page, and the ecosystem listings; wire
into the supply-chain, doc-coverage, and root test gates. Seamless DOM-adopting
hydration is a deferred future core RFC.
